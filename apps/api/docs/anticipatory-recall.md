# Anticipatory Recall Engine (ARE)

> *Memory that thinks ahead.*

## Overview

The Anticipatory Recall Engine (ARE) is an opt-in enhancement to Engram's recall system. When enabled, it runs alongside standard memory recall to surface **adjacent memories and insights the agent didn't explicitly ask for**.

Think of it like human intuition — you mention "Christmas" and your brain fires: family, travel plans, gifts. ARE brings that associative quality to AI agent memory.

## Quick Start

Add `anticipatory` to your recall request:

```json
POST /api/v1/recall
{
  "query": "How is the Engram database doing?",
  "userId": "beaux",
  "anticipatory": {
    "enabled": true
  }
}
```

Response includes anticipated memories alongside standard results:

```json
{
  "memories": [
    { "id": "mem_1", "raw": "Standard recall result..." },
    {
      "id": "mem_2",
      "raw": "Railway deployment uses shared DATABASE_URL...",
      "recallSource": "anticipatory",
      "anticipatory": {
        "strategy": "entity_radiation",
        "reason": "Related to Engram via entity: Railway",
        "salience": 0.68,
        "entityPath": ["Engram", "Railway"]
      }
    }
  ],
  "anticipatoryMeta": {
    "strategiesRun": ["entity_radiation", "insight_injection"],
    "latencyMs": 47,
    "circuitBreakerActive": false,
    "signals": {
      "entitiesDetected": ["Engram"],
      "topics": ["technical"]
    }
  }
}
```

## How It Works

### 1. Context Signal Extraction

When a recall request arrives, ARE extracts signals from the query text:

- **Entity detection** — matches query against known entity names/aliases from the knowledge graph
- **Topic detection** — keyword-based topic classification (family, health, projects, technical, schedule, work)
- **Temporal context** — time of day, day of week

This is pure computation — no database queries, <10ms.

### 2. Strategy Selection

Based on detected signals, ARE picks **at most 2 strategies** to run:

| Strategy | Trigger Signal | What It Does |
|---|---|---|
| **Entity Radiation** | Entity names in query | Traverses knowledge graph 1-hop, pulls memories from adjacent entities |
| **Insight Injection** | Topics or entities in query | Surfaces relevant waking cycle insights not yet shown |

Strategy selection is a lightweight scoring function — no DB hits.

### 3. Parallel Execution

Selected strategies run in parallel, each with a per-strategy timeout. Results are:
- Deduplicated against standard recall results
- Filtered by minimum salience score
- Ranked by salience descending
- Capped at `maxResults`

### 4. Safety: Circuit Breaker

ARE monitors its own latency. If p95 exceeds 200ms over a 5-minute window (10+ samples), ARE automatically disables itself for 2 minutes. Standard recall is **never** affected.

## Configuration

### API Options

```typescript
anticipatory?: {
  enabled?: boolean;        // default: false
  maxResults?: number;      // default: 3, max: 5
  strategies?: string[];    // override strategy selection
  minSalience?: number;     // default: 0.3 (0-1)
}
```

### Environment Variables

```bash
# Master toggle (must be 'true' to enable)
ANTICIPATORY_ENABLED=true

# Strategy toggles
ANTICIPATORY_ENTITY_RADIATION=true
ANTICIPATORY_INSIGHT_INJECTION=true

# Performance
ANTICIPATORY_LATENCY_BUDGET_MS=100
ANTICIPATORY_CIRCUIT_BREAKER_P95_MS=200
ANTICIPATORY_MAX_RESULTS=3
ANTICIPATORY_MAX_ENTITY_HOPS=1

# Feedback learning
ANTICIPATORY_EVENT_FLUSH_INTERVAL_MS=30000
ANTICIPATORY_MIN_SAMPLES_FOR_LEARNING=20
```

## Feedback Loop

ARE learns which anticipatory results are useful. Submit feedback via:

```json
POST /api/v1/anticipatory/feedback
{
  "memoryId": "mem_xxx",
  "recallId": "rcl_xxx",
  "wasUseful": true
}
```

Feedback adjusts per-user per-strategy weights. After 20+ samples, the system adapts strategy selection to each user's patterns.

## Strategies In Depth

### Entity Radiation

When your query mentions a known entity (e.g., "Engram"), ARE:

1. Looks up the entity in the knowledge graph
2. Traverses 1-hop relationships to find adjacent entities
3. For each adjacent entity, pulls the top memory (by effectiveScore × recency)
4. Applies diversity filter (max 1 memory per adjacent entity)
5. Computes salience from edge weight × memory score × recency decay

**Example:** Query mentions "Engram" → graph shows Railway, Prisma, pgvector → surfaces the most relevant memory about Railway's connection to Engram.

### Insight Injection

Surfaces INSIGHT-layer memories generated by the Waking Cycle:

1. Finds INSIGHT memories matching current topics/entities
2. Filters out recently-surfaced insights (4-hour cooldown)
3. Ranks by: relevance × confidence × freshness × surfacing-decay
4. Marks surfaced insights to avoid repetition

**Key:** ARE doesn't generate insights — the Waking Cycle does. ARE is the delivery mechanism that gets those insights in front of agents when they're relevant.

## Performance

| Metric | Target | Hard Limit |
|---|---|---|
| ARE total latency | <100ms | 200ms (circuit breaker) |
| DB queries per recall | ≤3 | 5 |
| LLM calls per recall | 0 | 0 |

ARE runs after standard recall completes and never blocks or slows it down.

## Database Models

ARE adds two tables:

- **`anticipatory_events`** — Buffered event log for feedback learning
- **`anticipatory_weights`** — Per-user per-strategy learned weights
